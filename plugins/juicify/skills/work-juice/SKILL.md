# Work Juice — 작업 경험에서 인사이트를 짜내다

이 세션에서 진행한 개선 작업을 케이스 스터디 리포트로 정리한다.

**Language**: 항상 사용자의 언어로 응답하고 리포트를 작성한다.

## 케이스 정의

**하나의 케이스** = "이슈 발견 → 원인 분석 → 개선 → 검증"이 하나의 사이클을 이루는 단위.
한 세션에 여러 케이스가 존재할 수 있다. 반드시 케이스 단위로 분리하여 리포팅한다.
여러 케이스를 하나의 리포트에 합치지 않는다.

## 실행 플로우

### Step 0: 인자 확인

- `$ARGUMENTS`가 "latest"이면 → Step 2로 직행
- `$ARGUMENTS`가 "select"이면 → Step 3으로 직행
- `$ARGUMENTS`가 비어있으면 → Step 1로

### Step 1: 모드 선택

사용자에게 질문한다:

> **최근 케이스만 리포팅할까요, 여러 케이스 중 선택할까요?**
> 1. 최근 케이스
> 2. 케이스 선택

- 1 선택 → Step 2
- 2 선택 → Step 3

### Step 2: 최근 케이스 확인 (단일)

세션에서 가장 최근에 완료된 케이스를 식별하고 한 줄 요약을 제시한다:

> **가장 최근 케이스:**
> "commands에서 skills로 전환 — 반복 프로세스를 전역 스킬로 구조화"
>
> 이 케이스를 리포팅할까요? (Y/N)

- Y → Step 4 (리포팅 진행)
- N → 그 이전 케이스를 제시하고 다시 확인. 반복.

### Step 3: 케이스 선택 (복수)

세션 전체를 분석하여 독립된 케이스를 추출하고 번호를 매겨 나열한다:

> **이 세션의 케이스 목록:**
> 1. v1 Python 파이프라인에서 v2 Claude Code 서브에이전트로 전환
> 2. source-reader 3축 필터링 확장 (기간/성격/파일지정)
> 3. 콘텐츠 품질 검수 체계 도입 (content-reviewer + 소재 선별 스킬)
> 4. 반복 프로세스를 전역 스킬로 전환 (commands → skills)
>
> 리포팅할 케이스 번호를 선택하세요 (복수 가능, 예: 1,3)

선택된 케이스 각각에 대해 Step 4를 반복한다. 각 케이스는 별도 MD 파일로 생성.

### Step 4: 리포팅 진행

선택된 케이스에 대해:
1. 해당 케이스의 세션 맥락만 추출 (다른 케이스 내용 혼입 금지)
2. 리포트 구조에 맞춰 작성
3. `docs/work-notes/YYYY-MM-DD-{주제-요약}.md`로 저장
4. 저장 경로와 요약을 사용자에게 알려줌

## 리포트 구조

```markdown
# {제목}

## 요약
(2~3문장으로 이 케이스만 요약)

---

## 1. 현상 발견
- 어떤 상황에서 무엇을 테스트/시도했는가
- 기대와 다른 어떤 결과가 나왔는가
- 왜 이것이 문제(또는 개선 기회)라고 판단했는가

## 2. 원인 분석
- 문제의 근본 원인은 무엇이었는가
- 원인 체인 (A → B → C → 문제)
- 기존 시스템에서 왜 이 문제를 잡지 못했는가 (해당되는 경우)

## 3. 개선 방향 논의
- 어떤 선택지를 고려했는가
- 각 선택지의 장단점
- 최종 결정과 그 이유

## 4. 구현
- 무엇을 어떻게 변경했는가 (핵심만)
- 변경의 설계 의도

## 5. 검증
- Before vs After 비교 (해당되는 경우)
- 정량/정성 결과

## 6. 레슨런
- 이 케이스에서 배운 것 (2~4개)
- 다른 상황에도 적용 가능한 일반화된 인사이트
```

## 작성 원칙

- **케이스 단위 분리 철저**: 하나의 리포트 = 하나의 케이스. 혼재 금지.
- **경험의 자산화**: 1순위 목적은 본인의 성장 기록. 나중에 다시 읽었을 때 사고 과정을 복기할 수 있어야 한다. 2순위는 동료와의 공유.
- 코드 레벨 디테일은 최소화. 아키텍처와 사고방식 중심으로 작성.
- "무엇을 했다"보다 "왜 그렇게 판단했다"에 비중을 둔다.
- 개인적으로 민감한 정보(구체적 회사명, 사적 감정 등)는 일반화하여 작성한다.
- 레슨런은 이 프로젝트에 국한되지 않는, 재사용 가능한 인사이트로 정리한다.
